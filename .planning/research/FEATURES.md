**领域:** PostgreSQL 云备份系统（Docker 容器 + Backblaze B2 云存储）
**调研日期:** 2026-04-05
**置信度:** HIGH（基于 PostgreSQL 官方文档、B2 官方文档、项目代码库分析）

## 特性总览

### Table Stakes（必备特性）

缺少这些特性，系统不可用于生产环境。用户不会因为你有这些而赞赏，但没有的话系统就是残缺的。

| # | 特性 | 为什么必备 | 复杂度 | 依赖 | 备注 |
|---|------|-----------|--------|------|------|
| TS-1 | 多数据库完整备份（pg_dump -Fc） | PostgreSQL 有多个独立数据库（keycloak_db、findclass_db），缺一个就意味着灾难恢复时数据不完整 | S | 无 | 用 -Fc 自定义格式，自带 zlib 压缩且支持并行恢复。必须对每个数据库单独运行 pg_dump |
| TS-2 | 全局对象备份（pg_dumpall --globals-only） | pg_dump 不备份角色和表空间定义。恢复时缺少角色导致权限错误，所有应用无法连接数据库 | S | TS-1 | 单独运行，输出角色+表空间定义。与数据库备份分开存储 |
| TS-3 | 备份文件命名规范（时间戳+数据库名） | 28 个备份文件（7天 x 每天4次）没有规范命名就无法识别和排序，清理逻辑也无法判断文件年龄 | S | TS-1 | 格式：`{db_name}_{YYYYMMDD_HHmmss}.dump`。使用 Pacific/Auckland 时区 |
| TS-4 | 自定义压缩格式（pg_dump -Fc） | 未压缩的纯 SQL 文件是数据库实际大小的 3-5 倍，增加存储成本和上传时间。-Fc 自带压缩无需额外处理 | S | TS-1 | -Fc 格式自带 zlib 压缩，不要再用 gzip 二次压缩。同时支持并行恢复 |
| TS-5 | 云存储自动上传 | 备份只存本地等于没有备份——服务器磁盘损坏则备份同时丢失 | M | TS-1, TS-3 | 使用 AWS CLI + B2 S3 兼容 API。上传后立即删除本地文件（节省磁盘空间） |
| TS-6 | 上传失败重试（指数退避） | 网络不稳定时上传中断很常见。没有重试机制会导致备份看似成功实则未上传，灾难时才发现没有云端备份 | S | TS-5 | 最多重试 3 次，间隔 10s/30s/90s。检查 AWS CLI 退出码 |
| TS-7 | 上传后校验和验证 | 网络传输可能静默损坏文件。不验证的话，恢复时才发现备份已损坏为时已晚 | S | TS-5 | 上传前计算 SHA256，上传后通过 B2 文件元数据对比。不匹配则标记为失败 |
| TS-8 | 应用层保留清理（7天策略） | B2 生命周期规则对时间戳命名的独立文件无效（PITFALLS #3）。没有清理则存储成本无限增长 | M | TS-3, TS-5 | 列出 bucket 中备份文件，删除超过 8 天的（7天+1天安全缓冲）。同时清理本地残留 |
| TS-9 | 一键恢复脚本 | 备份的最终目的是恢复。没有恢复脚本的备份系统在灾难时刻需要手动操作，出错概率极高 | M | TS-1, TS-5 | 从 B2 下载 -> 创建干净数据库（template0）-> pg_restore -> ANALYZE。必须包含完整的错误处理 |
| TS-10 | 备份成功/失败日志 | 没有日志就无法知道备份是否正常运行。结构化日志是所有后续监控和告警的基础 | S | 无 | 包含：时间戳、数据库名、文件大小、耗时、状态（成功/失败）、错误详情 |
| TS-11 | 凭证通过环境变量管理 | 硬编码密钥 = 代码仓库泄露 = 所有备份数据暴露（PITFALLS #4）。这是最常见的安全事故 | S | 无 | 所有凭证（B2 Key、DB 密码）通过环境变量传入。绝不硬编码。使用 Jenkins Credentials 管理 |
| TS-12 | 最低权限 API Key | Master Key 泄露影响整个 B2 账户。专用受限 Key 泄露只影响备份 bucket | S | TS-11 | 创建 B2 Standard Application Key：仅限备份 bucket + writeFiles + deleteFiles + listFiles + fileNamePrefix 限制 |
| TS-13 | 备份前健康检查 | PostgreSQL 容器可能已停止或网络不通。盲目执行 pg_dump 会产生空文件或错误文件，误报为"备份成功" | S | 无 | 先 `pg_isready` 验证连接，失败则立即退出并告警。不生成任何备份文件 |

### Important（重要特性）

没有这些特性，系统可以运行但存在显著风险。应该尽快实现。

| # | 特性 | 为什么重要 | 复杂度 | 依赖 | 备注 |
|---|------|-----------|--------|------|------|
| IM-1 | 备份完整性验证（pg_restore --list） | pg_dump 可能返回 0 但实际备份不完整或损坏（PITFALLS #8）。只有用 pg_restore --list 验证过才算真正成功 | S | TS-1 | 备份后立即运行 `pg_restore --list backup.dump`。失败则标记整个备份为失败，不上传 |
| IM-2 | 备份失败告警（Webhook） | 备份静默失败是最危险的——一切看起来正常，直到需要恢复时才发现没有备份 | M | TS-10, TS-11 | 失败时发送 Webhook 通知（Slack/Discord/企业微信）。包含：失败时间、数据库、错误信息 |
| IM-3 | 列出可用备份 | 恢复时需要知道有哪些备份可用。手动登录 B2 控制台查看效率低且容易出错 | S | TS-3, TS-5 | `restore.sh --list` 命令列出所有可用备份，按时间排序。显示：文件名、大小、日期、数据库 |
| IM-4 | 恢复指定数据库 | 恢复整个集群是不必要的风险。通常只需要恢复某个出问题的数据库 | S | TS-9, IM-3 | `restore.sh --database findclass_db --timestamp 20260405_120000`。不影响其他运行中的数据库 |
| IM-5 | 恢复到不同数据库名（测试恢复） | 直接恢复到生产数据库进行测试是灾难性行为。需要安全地验证备份是否可用 | M | TS-9, IM-4 | `restore.sh --database findclass_db --target-db findclass_db_test`。恢复到临时数据库用于验证 |
| IM-6 | 每周自动恢复测试 | 备份从未被恢复过 = 备份可能不可用。每周自动测试是验证备份有效性的唯一可靠方法（PITFALLS #2） | L | TS-9, IM-5, IM-1 | Jenkins 定时任务：下载最新备份 -> 恢复到临时数据库 -> 验证表/数据完整性 -> 清理。失败时告警 |
| IM-7 | 备份持续时间追踪 | 如果备份从 5 分钟增长到 60 分钟，说明数据库在快速增长或出了问题。需要基线对比 | S | TS-10 | 记录每次备份的开始/结束时间。与最近 7 天平均值对比，偏差 >50% 时输出警告 |
| IM-8 | 磁盘空间检查 | 磁盘空间不足时 pg_dump 可能写入不完整文件但不报错（PITFALLS #8）。备份前检查磁盘空间是必要的预防措施 | S | 无 | 备份前检查可用空间 >= 预估备份大小 x 2。空间不足时跳过并告警 |
| IM-9 | 未完成上传清理 | B2 的 unfinished large files 默认不清理，持续计费（PITFALLS #6）。需要在应用层处理 | S | TS-5, TS-8 | 设置 B2 bucket 生命周期规则：`daysFromStartingToCancelingUnfinishedLargeFiles: 1`。在清理逻辑中也处理残留上传 |
| IM-10 | 结构化错误处理和退出码 | 脚本没有统一退出码，监控系统无法判断备份是否成功。Jenkins 需要标准退出码来标记构建状态 | S | 无 | 定义标准退出码：0=成功、1=连接失败、2=备份失败、3=上传失败、4=清理失败、5=验证失败 |

### Nice-to-Have（锦上添花）

这些特性提升体验但不是关键路径。可以在核心功能稳定后逐步添加。

| # | 特性 | 价值 | 复杂度 | 依赖 | 备注 |
|---|------|------|--------|------|------|
| NH-1 | 存储用量追踪 | 了解备份存储成本趋势，优化保留策略 | S | TS-10 | 每次备份后记录 B2 bucket 总大小。周报输出存储趋势 |
| NH-2 | 带宽限速 | 避免备份上传影响生产服务网络性能 | M | TS-5 | AWS CLI `--max-rate` 参数。对当前规模可能不需要（数据库小，上传快） |
| NH-3 | 分片上传 | 大文件上传更稳定，断点续传 | M | TS-5 | AWS CLI 自动处理 >8MB 文件的分片上传。当前数据库规模可能不需要手动配置 |
| NH-4 | 上传进度日志 | 长时间上传时了解进度，避免误判为挂起 | S | TS-5 | AWS CLI `--progress` 参数。配合 Jenkins 日志输出 |
| NH-5 | 备份文件加密（客户端） | 防止 B2 Access Key 泄露后攻击者读取备份数据 | M | TS-5 | 使用 GPG 或 age 加密后再上传。增加恢复复杂度。当前 SSE-B2 + 严格 Key 权限已足够 |
| NH-6 | 恢复后数据完整性校验 | 恢复后对比行数/表数，确认数据完整 | M | IM-6 | 比较源库和恢复库的表数量、每表行数。适合自动化恢复测试流程 |
| NH-7 | 备份指标仪表盘 | 可视化备份历史、趋势、存储使用 | L | TS-10, IM-7 | 需要 Prometheus/Grafana 或类似方案。当前规模用日志即可 |
| NH-8 | Webhook 成功通知 | 每次备份成功都收到通知 | S | TS-10, IM-2 | 容易实现但通知噪音高。建议仅失败时通知，成功时只在日志中记录 |
| NH-9 | 多云存储冗余 | 备份同时上传到两个云存储服务 | M | TS-5 | 增加成本和复杂度。当前 B2 单一存储已足够可靠 |
| NH-10 | 备份文件预览（pg_restore --list 输出存档） | 不下载备份就能查看内容概要 | S | TS-1, IM-1 | 将 pg_restore --list 输出作为元数据一起上传。调试时有用但非必需 |

### Anti-Features（明确不做的特性）

这些特性看起来有用，但实际上会增加复杂度、成本或风险，或者与项目需求不匹配。

| # | 特性 | 为什么被要求 | 为什么不应该做 | 替代方案 |
|---|------|-------------|---------------|---------|
| AF-1 | 增量备份（pg_basebackup + WAL） | 看似节省存储和时间 | PostgreSQL 完整备份对当前数据库规模已足够快。增量备份需要管理 WAL 归档链，恢复流程复杂度高，容错率低。对于小型数据库，增量备份节省的空间不值得增加的复杂度 | 保持 pg_dump -Fc 完整备份，简单可靠 |
| AF-2 | 实时复制/流备份 | 最高级别的数据保护 | 需要主从架构，至少双倍基础设施成本。当前项目是小型生产环境，6-12 小时 RPO 完全满足需求。实时复制是大型企业或金融场景的需求 | pg_dump 定时备份 + 6-12 小时 RPO |
| AF-3 | PITR 时间点恢复 | 理论上能恢复到任意时间点 | 需要基础备份 + 完整 WAL 归档链，复杂度极高。恢复步骤多，出错概率大。当前已有 WAL 归档配置但未配合基础备份，形成了技术债务（PITFALLS #9） | 使用 pg_dump 多频率备份实现粗粒度时间点恢复（每 6 小时一个恢复点） |
| AF-4 | 跨区域备份复制 | 地理灾难恢复 | 增加存储成本（至少翻倍）和管理复杂度。B2 本身已有数据中心冗余。对于新西兰小型生产环境，单区域存储足够 | B2 单区域存储 + 本地短期备份 |
| AF-5 | 手动备份触发 UI | 方便运维手动触发 | 增加一个 Web UI 的开发和维护成本。Jenkins 已提供手动触发能力。Web UI 还需要额外的认证和授权 | 通过 Jenkins 手动触发备份任务，或命令行 `./backup.sh --now` |
| AF-6 | 自定义加密方案 | 更高的安全性幻觉 | 自定义加密容易出错（密钥管理、加密库选择、IV 处理），反而降低安全性。B2 SSE-B2 已提供存储加密。如果需要更高安全性，使用 SSE-C（客户管理密钥）而非自行实现加密 | B2 SSE-B2（默认）+ 严格 API Key 权限管理 |
| AF-7 | 备份压缩格式选择（gzip/zstd/lz4） | 灵活性 | pg_dump -Fc 已内置 zlib 压缩，效果足够好。提供多种压缩格式选择增加测试矩阵（每种格式都需要测试备份+恢复流程），没有实际收益 | 统一使用 -Fc 格式，不提供压缩选项 |
| AF-8 | 数据库自动发现 | 动态发现所有数据库并备份 | 自动发现会备份 template0/template1 等系统数据库，浪费空间。如果新创建了数据库但忘记配置备份参数（用户名等），可能备份不完整 | 在配置文件中明确列出需要备份的数据库列表，新增数据库时手动添加（更安全） |
| AF-9 | 备份文件保留策略可配置（保留天数/数量） | 灵活性 | 增加配置复杂度和测试负担。7天保留策略是项目约束中已明确的。过早提供配置选项意味着需要测试多种组合 | 硬编码 7 天保留策略。如需调整，直接修改配置文件中的保留天数常量 |

### 依赖说明

- **TS-1 是核心节点：** 多数据库备份是所有其他特性的基础。没有备份就没有上传、没有恢复、没有测试
- **TS-9 恢复脚本是第二核心：** 恢复测试（IM-6）依赖恢复脚本。恢复脚本依赖上传功能（需要从云端下载）
- **TS-10 日志是横切关注点：** 几乎所有特性都依赖日志，但日志本身独立于备份流程
- **IM-6 是最复杂特性：** 恢复测试依赖完整的备份+上传+恢复链路，是系统集成的最终验证
- **TS-13 健康检查是第一道防线：** 必须最先执行，避免在数据库不可用时浪费时间和空间

## MVP 定义

### Launch With（v1 -- 核心备份+上传）

最小可用产品。能自动备份、上传、清理、恢复。

- [ ] **TS-13 健康检查** -- 避免在数据库不可用时执行备份
- [ ] **TS-1 多数据库备份** -- 核心功能，逐一备份 keycloak_db 和 findclass_db
- [ ] **TS-2 全局对象备份** -- 与数据库备份一起，备份角色和表空间
- [ ] **TS-3 命名规范** -- 时间戳+数据库名，支持后续清理和恢复
- [ ] **TS-4 -Fc 格式** -- 压缩+并行恢复支持
- [ ] **TS-10 结构化日志** -- 所有必要信息：时间、库名、大小、耗时、状态
- [ ] **TS-11 凭证管理** -- 环境变量，不硬编码
- [ ] **TS-12 最低权限 Key** -- 受限 B2 Application Key
- [ ] **IM-1 备份验证** -- pg_restore --list 验证备份完整性
- [ ] **IM-8 磁盘空间检查** -- 备份前检查可用空间
- [ ] **IM-10 统一退出码** -- 支持监控系统判断状态
- [ ] **TS-5 云存储上传** -- 上传到 B2，上传后删除本地文件
- [ ] **TS-6 上传失败重试** -- 指数退避，最多 3 次
- [ ] **TS-7 校验和验证** -- 上传后验证文件完整性
- [ ] **TS-8 保留清理** -- 应用层删除超过 8 天的备份
- [ ] **IM-9 未完成上传清理** -- 清理 B2 残留上传
- [ ] **TS-9 一键恢复脚本** -- 完整恢复流程：下载->建库->恢复->ANALYZE
- [ ] **IM-3 列出可用备份** -- 查看可用备份列表
- [ ] **IM-4 恢复指定数据库** -- 恢复单个数据库

### Add After Validation（v1.x -- 监控和测试）

核心备份验证可用后，添加监控和自动化测试。

- [ ] **IM-2 备份失败告警** -- Webhook 通知，失败时立即知道
- [ ] **IM-5 恢复到测试库** -- 安全验证备份可用性
- [ ] **IM-6 每周自动恢复测试** -- Jenkins 定时任务验证恢复流程
- [ ] **IM-7 备份耗时追踪** -- 监控备份性能趋势
- [ ] **NH-4 上传进度日志** -- Jenkins 日志中显示上传进度

### Future Consideration（v2+ -- 优化和增强）

系统稳定运行后再考虑。

- [ ] **NH-1 存储用量追踪** -- 成本监控
- [ ] **NH-6 恢复后数据完整性校验** -- 行数/表数对比
- [ ] **NH-5 客户端加密** -- 更高安全要求时
- [ ] **NH-10 备份预览** -- 调试辅助

## 竞品/工具特性对比

| 特性 | pg_dump 脚本（本项目） | pgBackRest | Barman | pg_probackup |
|------|----------------------|------------|--------|-------------|
| 完整备份 | pg_dump -Fc | 增量+完整 | 增量+完整 | 增量+完整 |
| 备份格式 | 自定义二进制 | 专用压缩格式 | 专用格式 | 专用格式 |
| 多数据库 | 逐一 pg_dump | 集群级（物理） | 集群级（物理） | 集群级（物理） |
| 云存储 | AWS CLI + B2 S3 | 原生 S3 支持 | S3/SSH | 本地为主 |
| 恢复粒度 | 单数据库/单表 | 集群级 | 集群级 | 集群级 |
| 并行恢复 | pg_restore -j | 原生并行 | 原生并行 | 原生并行 |
| PITR | 不支持 | 支持 | 支持 | 支持 |
| 复杂度 | LOW（脚本） | HIGH（配置多） | HIGH | MEDIUM |
| Docker 兼容 | HIGH（标准工具） | MEDIUM（需要配置） | LOW（不适合） | MEDIUM |
| 维护成本 | LOW | MEDIUM | HIGH | MEDIUM |

**本项目选择 pg_dump 脚本的理由：** 数据库规模小、Docker 环境、不需要 PITR、不需要增量备份。pg_dump 是 PostgreSQL 自带工具，无额外依赖，与 Docker 环境天然兼容。pgBackRest/Barman 是为大规模集群设计的，对当前项目来说是过度工程。

## 特性与 PITFALLS 对应关系

| PITFALL | 对应特性 | 如何解决 |
|---------|---------|---------|
| #1 多数据库遗漏 | TS-1, TS-2 | 显式列出所有数据库，逐一备份 + pg_dumpall --globals-only |
| #2 恢复失败 | TS-4, TS-9, IM-1, IM-6 | -Fc 格式 + 完整恢复脚本 + 备份验证 + 自动恢复测试 |
| #3 B2 生命周期规则误配 | TS-8, IM-9 | 应用层清理（不依赖 B2 规则） + 未完成上传清理 |
| #4 凭证硬编码 | TS-11, TS-12 | 环境变量 + 受限 B2 Key |
| #5 Docker 调度失效 | TS-10 | Jenkins cron 调度 + 结构化日志 |
| #6 网络传输失败 | TS-6, TS-7 | 重试机制 + 校验和验证 |
| #7 未压缩备份 | TS-4 | pg_dump -Fc 自带压缩 |
| #8 监控盲区 | IM-1, IM-8, IM-10 | 备份验证 + 磁盘检查 + 统一退出码 |
| #9 WAL 归档混淆 | AF-3 | 明确不用 PITR，评估关闭 archive_mode |
| #10 加密误解 | TS-12, AF-6 | SSE-B2 + 严格 Key 权限（不自行实现加密） |
