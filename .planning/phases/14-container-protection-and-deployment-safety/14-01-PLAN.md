  <action>
将 noda-ops 容器从 root 运行改为 nodaops 用户运行。参考 Dockerfile.findclass-ssr 中的 addgroup/adduser 模式。

**deploy/Dockerfile.noda-ops 修改：**
1. 在 `WORKDIR /app` 之后、`COPY scripts/backup/` 之前，添加：
   ```dockerfile
   # 创建非 root 用户
   RUN addgroup -S nodaops && adduser -S -G nodaops nodaops
   ```
2. 修改 crontab COPY 目标：
   ```dockerfile
   COPY deploy/crontab /etc/crontabs/nodaops
   ```
   （原来是 `/etc/crontabs/root`）
3. 在 `RUN mkdir -p ...` 那行之后，添加 chown：
   ```dockerfile
   RUN chown -R nodaops:nodaops /app /var/log/supervisor /var/log/noda-backup /app/history /run/supervisor
   RUN mkdir -p /home/nodaops/.config/rclone && chown -R nodaops:nodaops /home/nodaops
   ```
4. 在 HEALTHCHECK 之前添加：
   ```dockerfile
   USER nodaops
   ```

**deploy/entrypoint-ops.sh 修改：**
1. 将 `mkdir -p /root/.config/rclone` 改为 `mkdir -p /home/nodaops/.config/rclone`
2. 将 `cat > /root/.config/rclone/rclone.conf <<EOF` 改为 `cat > /home/nodaops/.config/rclone/rclone.conf <<EOF`
3. 在 rclone 配置写入之前添加环境变量：
   ```bash
   export RCLONE_CONFIG=/home/nodaops/.config/rclone/rclone.conf
   ```
4. 将 `crontab -l` 改为 `crontab -l -u nodaops 2>/dev/null || echo "无定时任务"`（注意：dcron 不需要 -u 参数，而是直接读取 /etc/crontabs/nodaops 文件，所以保留 `crontab -l` 但确保它以 nodaops 用户身份运行即可）
5. 移除 `sed -i 's/autostart=true/autostart=false/'` 行对 supervisord.conf 的修改（因为以 nodaops 用户运行时可能没有写权限，改用环境变量控制或提前在 Dockerfile 中处理）

**deploy/supervisord.conf 修改：**
1. 将 `[program:cron]` 段的 `environment=HOME="/root",PATH=...` 改为：
   ```
   environment=HOME="/home/nodaops",PATH="/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin"
   ```
2. 将 `[program:cloudflared]` 段的 `environment=HOME="/root"` 改为：
   ```
   environment=HOME="/home/nodaops"
   ```
3. 将 `pidfile=/var/run/supervisord.pid` 改为：
   ```
   pidfile=/run/supervisor/supervisord.pid
   ```
   （确保 pidfile 写入 tmpfs 可写路径）

**deploy/crontab 修改：**
无需修改内容。crontab 文件内容本身不包含用户信息，dcron 通过文件路径 `/etc/crontabs/nodaops` 确定所属用户。

**在现有 `postgres:` 服务段落内追加（在 volumes 之后）：**
```yaml
    security_opt:
      - no-new-privileges:true
    cap_drop:
      - ALL
    cap_add:
      - CHOWN
      - DAC_OVERRIDE
      - FOWNER
      - SETGID
      - SETUID
    read_only: true
    tmpfs:
      - /var/run/postgresql
      - /tmp
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"
    stop_grace_period: 30s
```

**在现有 `keycloak:` 服务段落内追加（在 healthcheck 之后）：**

      options:
        max-size: "10m"
        max-file: "3"

注意：Keycloak 26.x 已默认以 uid 1000 非 root 运行，不需要 cap_drop/cap_add。

**在现有 `findclass-ssr:` 服务段落内追加（在 deploy 之后）：**

**新增或追加 nginx 服务段落：**

如果 Plan 02 已添加 nginx 段（含 errors volume），则在该段中追加安全选项。如果未添加，新建以下段落：

**新增 noda-ops 服务段落：**
```yaml
  # ----------------------------------------
  # Noda Ops 安全加固（生产环境）
  # ----------------------------------------
  noda-ops:
    security_opt:
      - no-new-privileges:true
    cap_drop:
      - ALL
    cap_add:
      - CHOWN
      - DAC_OVERRIDE
      - FOWNER
      - SETGID
      - SETUID
    read_only: true
    tmpfs:
      - /tmp
      - /var/log/supervisor
      - /run/supervisor
      - /home/nodaops
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"
    stop_grace_period: 30s
```

**关键注意事项：**
- noda-ops 的 tmpfs 包含 `/home/nodaops` 用于 rclone config 的运行时写入
- noda-ops 的 `/app/history` 通过 volume `./volumes/history:/app/history` 挂载，host volume 会覆盖 tmpfs，不需要 tmpfs 条目
- noda-ops 的 `/tmp/postgres_backups` 通过 volume `./volumes/backup:/tmp/postgres_backups` 挂载，不需要 tmpfs
- `/var/log/noda-backup` 通过 volume `./volumes/logs:/var/log/noda-backup` 挂载，不需要 tmpfs
- postgres 的 cap_add 包含 CHOWN, DAC_OVERRIDE, FOWNER, SETGID, SETUID — 这些是 postgres 官方镜像需要的
- nginx 仅需 NET_BIND_SERVICE 用于绑定 80 端口
- 所有 tmpfs 都是不持久化的 — 容器重启后 tmpfs 内容丢失，但持久化数据通过 volumes 保留

验证：运行 `docker compose -f docker/docker-compose.yml -f docker/docker-compose.prod.yml config --quiet` 确保无语法错误。

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| 容器 -> 宿主机内核 | 容器内进程通过系统调用访问内核功能（cap_drop/cap_add 限制此边界） |
| 容器 -> 文件系统 | read_only + tmpfs 限制容器内文件写入范围 |
| 容器 -> 网络 | no-new-privileges 防止权限提升攻击 |
| 日志 -> 磁盘 | json-file + max-size/max-file 防止日志占满磁盘 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-01 | Privilege Escalation | 容器逃逸 via root | mitigate | non-root user (nodaops) + no-new-privileges:true + cap_drop: ALL |
| T-14-02 | Privilege Escalation | 过多内核 capabilities | mitigate | cap_drop: ALL + 按需白名单 cap_add |
| T-14-03 | Denial of Service | 日志磁盘耗尽 | mitigate | json-file driver + max-size: 10m + max-file: 3 |
| T-14-04 | Denial of Service | 容器内文件系统写入失控 | mitigate | read_only: true + tmpfs 限定写入路径 |
| T-14-05 | Denial of Service | 优雅关闭超时数据丢失 | mitigate | stop_grace_period: 30s + backup trap 信号处理 |
</threat_model>
