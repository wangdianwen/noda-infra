**函数 1: http_health_check(container_name, [max_retries], [interval])**
per D-01/D-02：通过 `docker exec` 在目标容器内执行 wget 检测 `http://localhost:3001/api/health`。
- 使用 BusyBox wget：`docker exec "$container" wget --quiet --tries=1 --spider "http://localhost:3001/api/health"`
- 默认 30 次重试，间隔 4 秒（总超时 120 秒，覆盖 SSR 冷启动）
- 成功时 log_success 并 return 0
- 全部失败时 log_error + 打印最后 20 行容器日志 `docker logs "$container" --tail 20`，return 1
- 注意：`docker exec` 命令在 `if` 条件中使用时 `set -e` 不触发，必须用 `if ! ... then exit 1; fi` 显式处理

**函数 2: e2e_verify(target_env, [max_retries], [interval])**
- 获取容器名：`get_container_name "$target_env"` -> `findclass-ssr-{color}`
- 通过 nginx 容器 curl 目标容器：
  ```bash
  http_code=$(docker exec "$NGINX_CONTAINER" \
    curl -s -o /dev/null -w "%{http_code}" \
    "http://${container_name}:3001/api/health" 2>/dev/null || echo "000")
  ```
- 判断 HTTP 200 为成功
- 默认 5 次重试，间隔 2 秒
- 如果 nginx 容器没有 curl（ASSUMPTION A1），备选方案：`docker exec "$NGINX_CONTAINER" wget -q --spider "http://${container_name}:3001/api/health"`
- 实现时优先使用 curl，如果 curl 不存在则回退到 wget。使用 `docker exec "$NGINX_CONTAINER" which curl` 检测
- 全部失败时 log_error，return 1

**函数 3: cleanup_old_images(keep_count)**
per D-09：保留最近 N 个带标签的镜像（默认 N=5）。
- 列出所有 findclass-ssr 镜像（排除 latest 和 dangling）：`docker images findclass-ssr --format '{{.Repository}}:{{.Tag}}' | grep -v latest | sort`
- 如果镜像数超过 N，删除最旧的（按创建时间排序）
- 使用 `docker rmi` 删除，忽略删除失败（镜像可能被容器引用）
- 只打印日志，不中断部署流程

**主函数 main([apps_dir])：**

1. **前置检查**
   - 验证 Docker daemon 可用：`docker info >/dev/null 2>&1`
   - 验证 nginx 容器运行中：`is_container_running "$NGINX_CONTAINER"` = "true"
   - 验证网络存在：`docker network inspect "$NETWORK_NAME" >/dev/null 2>&1`

2. **读取活跃环境**
   ```bash
   local active_env target_env
   active_env=$(get_active_env)
   target_env=$(get_inactive_env)
   ```
   per D-04：自动检测，用户无需指定。

3. **获取代码目录和 SHA**
   per D-05：`apps_dir` 参数默认为当前目录 `.`。
   ```bash
   local apps_dir="${1:-.}"
   local short_sha
   short_sha=$(git -C "$apps_dir" rev-parse --short HEAD)
   ```
   验证 SHA 非空（防止不在 git 仓库中执行）。

4. **构建镜像**（per D-11/PIPE-03）
   ```bash
   log_info "步骤 1/7: 构建镜像"
   docker compose -f "$COMPOSE_FILE" build findclass-ssr
   ```
   per D-12：`set -e` 保证构建失败时脚本立即中止。

5. **添加 SHA 标签**（per D-10/PIPE-02）
   ```bash
   log_info "步骤 2/7: 标记镜像 findclass-ssr:${short_sha}"
   docker tag findclass-ssr:latest "findclass-ssr:${short_sha}"
   ```

6. **停止旧目标容器（如果存在）**
   ```bash
   local target_container
   target_container=$(get_container_name "$target_env")
   if [ "$(is_container_running "$target_container")" = "true" ]; then
     log_info "停止旧目标容器: $target_container"
     docker stop -t 30 "$target_container"
     docker rm "$target_container"
   fi
   ```

7. **启动新容器**
   ```bash
   log_info "步骤 3/7: 启动新容器 $target_container (镜像: findclass-ssr:${short_sha})"
   run_container "$target_env" "findclass-ssr:${short_sha}"
   ```

8. **HTTP 健康检查**（per D-01/D-02/TEST-03）
   ```bash
   log_info "步骤 4/7: HTTP 健康检查"
   if ! http_health_check "$target_container" "$HEALTH_CHECK_MAX_RETRIES" "$HEALTH_CHECK_INTERVAL"; then
     log_error "健康检查失败，保持当前环境: $active_env"
     log_warn "新容器 $target_container 仍在运行，可手动检查后停止"
     exit 1
   fi
   ```
   per TEST-05：健康检查失败时不切换流量、不停旧容器。

9. **切换流量**
   ```bash
   log_info "步骤 5/7: 切换流量 $active_env -> $target_env"
   update_upstream "$target_env"
   if ! docker exec "$NGINX_CONTAINER" nginx -t; then
     log_error "nginx 配置验证失败，回滚 upstream"
     update_upstream "$active_env"
     exit 1
   fi
   reload_nginx
   set_active_env "$target_env"
   ```

10. **E2E 验证**（per TEST-04）
    ```bash
    log_info "步骤 6/7: E2E 验证"
    if ! e2e_verify "$target_env" "$E2E_MAX_RETRIES" "$E2E_INTERVAL"; then
      log_error "E2E 验证失败，执行自动回滚"
      # 自动回滚到旧环境
      update_upstream "$active_env"
      docker exec "$NGINX_CONTAINER" nginx -t
      reload_nginx
      set_active_env "$active_env"
      log_warn "已回滚到 $active_env，新容器 $target_container 仍在运行"
      exit 1
    fi
    ```

11. **清理旧镜像**（per D-09）
    ```bash
    log_info "步骤 7/7: 清理旧镜像"
    cleanup_old_images 5
    ```

12. **完成**
    ```bash
    log_success "=========================================="
    log_success "蓝绿部署完成"
    log_success "=========================================="
    log_success "$active_env -> $target_env (镜像: findclass-ssr:${short_sha})"
    ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| 脚本参数 -> docker 命令 | 用户输入的 apps_dir 路径注入到 docker compose -f 命令 |
| git 输出 -> docker tag | git rev-parse 输出用于 docker tag 命令 |
| docker exec -> 容器内部 | 脚本通过 docker exec 在容器内执行 wget/curl |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-22-01 | Tampering | docker compose build 参数 | accept | apps_dir 路径由管理员手动指定，非外部输入；set -euo pipefail 防止意外扩展 |
| T-22-02 | Tampering | docker tag SHA 标签 | accept | SHA 来自 git rev-parse（本地仓库），不受外部输入控制；脚本验证 SHA 非空 |
| T-22-03 | Tampering | upstream 配置文件 | mitigate | update_upstream 使用原子写入（tmpfile + mv），已有模式；nginx -t 验证配置正确性 |
| T-22-04 | Information Disclosure | 环境变量文件 | accept | env 文件使用 tmpfs（/tmp），用后即删（run_container 函数内 rm -f） |
| T-22-05 | Denial of Service | nginx reload 失败 | mitigate | reload 前执行 nginx -t 验证；失败时回滚 upstream 到旧值；reload_nginx 函数检查容器运行状态 |
</threat_model>
