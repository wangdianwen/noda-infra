## 🔧 技术实现亮点

### 1. 多层验证机制

**Layer 1: 文件完整性**
- 使用 `pg_restore --list` 验证备份文件可读性
- 确保文件格式正确且未损坏

**Layer 2: SHA-256 校验和**
- 计算备份文件的 SHA-256 哈希值
- 与 B2 元数据中的校验和对比

**Layer 3: 数据结构验证**
- 检查恢复的数据库中的表数量
- 确保至少包含 1 个表（可配置最小值）

**Layer 4: 数据完整性验证**
- 检查第一个有数据的表的记录数
- 确保数据成功恢复

### 2. 安全的测试数据库管理

**命名规范**:
- 测试数据库必须以 `test_restore_` 前缀开头
- 删除前验证数据库名称，防止误删生产数据库

**冲突处理**:
- 检测数据库是否已存在
- 如果存在，先删除再重建
- 失败时保留数据库供调试

### 3. 可靠的下载机制

**重试策略**:
- 最多重试 3 次
- 指数退避：1s, 2s, 4s
- 每次重试前记录日志

**错误处理**:
- 下载失败返回明确的退出码
- 输出详细的错误信息
- 支持手动干预和调试

### 4. 完善的清理机制

**正常清理**:
- 测试成功后立即删除临时数据库
- 删除下载的备份文件
- 清理临时目录

**异常清理**:
- trap EXIT 确保任何情况下都执行清理
- 超时后强制清理（kill + 删除）
- 失败时保留资源供调试

### 5. 灵活的配置系统

**环境变量支持**:
- `TEST_DATABASES` - 测试数据库列表
- `TEST_TIMEOUT` - 超时时间
- `LOG_LEVEL` - 日志级别

**命令行参数**:
- 支持覆盖默认配置
- 提供帮助信息
- 参数验证和错误提示
