## Phase 4 需求分析

### 核心需求（VERIFY-02）

> 系统每周自动执行恢复测试，在临时数据库中验证备份文件可完整恢复，失败时发出告警。

### 用户期望

1. **自动化**：无需人工干预，每周自动执行
2. **可靠性**：多层验证确保备份可用
3. **安全性**：不影响生产数据库，使用临时库测试
4. **可观测性**：详细日志和明确的错误信息
5. **资源管理**：测试后自动清理，不留垃圾

### 技术约束

1. **依赖关系**：必须等待 Phase 3 完成
2. **执行时间**：整个流程需在 1 小时内完成
3. **网络连接**：需要访问 Backblaze B2 API
4. **数据库权限**：需要创建/删除临时数据库的权限
5. **磁盘空间**：需要足够空间存储下载的备份文件

## 关键技术决策

### 1. 测试架构

```
┌─────────────────┐
│  Scheduler      │
│  (Jenkins/cron) │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  Docker Container│
│  - PostgreSQL   │
│  - rclone       │
│  - Test Script  │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  Test Flow      │
│  1. Download    │
│  2. Restore     │
│  3. Verify      │
│  4. Cleanup     │
└─────────────────┘
```

**优势**：
- 隔离测试环境，不影响主系统
- 包含所有必要工具
- 便于调试和维护

### 2. 验证流程

```
下载备份文件
    │
    ▼
验证 SHA256 校验和 ─────失败────► 退出码 1
    │
    ▼
恢复到临时数据库 ─────────失败────► 退出码 2
    │
    ▼
验证文件可读性 ──────────失败────► 退出码 3
    │
    ▼
验证表数量 ─────────────失败────► 退出码 3
    │
    ▼
验证记录存在 ───────────失败────► 退出码 3
    │
    ▼
清理临时资源 ───────────失败────► 退出码 4
    │
    ▼
成功 ✅ (退出码 0)
```

### 3. 失败处理策略

| 失败阶段 | 退出码 | 处理方式 |
|----------|--------|----------|
| 下载失败 | 1 | 保留日志，标记失败 |
| 恢复失败 | 2 | 保留临时数据库和日志 |
| 验证失败 | 3 | 保留临时数据库和日志 |
| 清理失败 | 4 | 标记失败，记录警告 |
| 超时 | 5 | 强制清理，标记失败 |

**日志保留策略**：
- 成功：保留 7 天
- 失败：保留 30 天

## 关键技术挑战

### 1. 临时数据库管理

**挑战**：确保临时数据库不与生产库冲突，测试后正确清理

**解决方案**：
- 严格命名规范：`test_restore_{db_name}`
- 清理前验证：检查数据库名称前缀
- 超时保护：1 小时后强制清理

### 2. 验证完整性

**挑战**：如何快速且准确地验证恢复的数据是否完整

**解决方案**：
- 多层验证：文件 → 校验和 → 表结构 → 数据记录
- 关键记录检查：任意表记录数 > 0
- 可配置验证：支持自定义 SQL 查询

### 3. 资源管理

**挑战**：测试过程中可能创建多个临时文件和数据库

**解决方案**：
- 测试前检查：磁盘空间（数据库大小 × 2）
- 清理脚本：无论成功/失败都尝试清理
- 超时处理：强制终止并清理

### 4. 错误诊断

**挑战**：失败时需要快速定位问题

**解决方案**：
- 详细日志：记录每个步骤的时间戳和结果
- 分阶段退出码：明确标识失败阶段
- 失败保留：保留临时数据库和日志供调试
