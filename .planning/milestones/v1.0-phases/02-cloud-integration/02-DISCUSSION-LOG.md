## 决策 01: 云存储提供商选择

**日期:** 2026-04-06
**状态:** ✅ 已确定

### 决策内容
选择 **Backblaze B2** 作为云存储提供商

### 理由
1. **成本优势**: $0.005/GB/月，比 S3 标准（$0.023）便宜 4.6 倍
2. **11个9持久性**: 比 S3 的 11个9 更可靠
3. **原生兼容**: rclone 原生支持 B2，无需额外配置
4. **简单透明**: 定价清晰，无隐藏费用

### 影响
- 需要创建 Backblaze 账户和 bucket
- 需要生成 B2 Application Key
- 所有脚本基于 B2 API 开发

### 替代方案（未采用）
- ❌ **AWS S3**: 成本高 4.6 倍，对于备份场景不划算
- ❌ **Wasabi**: 虽然便宜（$0.0059），但不如 B2 成熟，rclone 支持不如 B2 完善
- ❌ **Google Cloud Storage**: 成本高（$0.02），且不提供免费额度

## 决策 02: 同步工具选择

**日期:** 2026-04-06
**状态:** ✅ 已确定

### 决策内容
使用 **rclone** 作为云存储同步工具

### 理由
1. **原生支持**: rclone 原生支持 B2，无需额外配置
2. **成熟稳定**: 广泛使用，社区活跃，bug 少
3. **内置重试**: 内置重试和断点续传机制
4. **校验和验证**: 支持 --checksum 验证文件完整性
5. **简单易用**: 配置简单，命令行友好

### 影响
- 需要在服务器上安装 rclone
- 所有云操作基于 rclone 命令
- 依赖 rclone 的退出码判断成功/失败

### 替代方案（未采用）
- ❌ **B2 CLI**: 官方 CLI，但功能不如 rclone 丰富，无校验和验证
- ❌ **AWS CLI + S3 兼容层**: 虽然可行，但增加复杂度，且 B2 不是完全兼容 S3 API
- ❌ **自定义 Python 脚本**: 开发成本高，需要维护，不如成熟工具

## 决策 03: 凭证管理方案

**日期:** 2026-04-06
**状态:** ✅ 已确定

### 决策内容
使用 **环境变量 + 临时配置文件** 管理凭证

### 理由
1. **安全性**: 凭证不在脚本中硬编码
2. **灵活性**: 不同环境可以使用不同的凭证
3. **兼容性**: 与现有 .env.backup 配置方式一致
4. **临时性**: rclone 配置文件在使用后立即删除

### 影响
- 扩展 .env.backup 文件，添加 B2 配置
- 每次上传时创建临时 rclone 配置文件
- 上传完成后立即删除配置文件

### 实现方案
```bash
# .env.backup
B2_ACCOUNT_ID=your_account_id
B2_APPLICATION_KEY=your_application_key
B2_BUCKET_NAME=noda-backups
B2_PATH=backups/postgres/
```

```bash
# 临时配置文件
rclone_config=$(mktemp)
chmod 600 "$rclone_config"
rclone config create b2remote backblazeb2 \
  --b2-account-id "$B2_ACCOUNT_ID" \
  --b2-account-key "$B2_APPLICATION_KEY" \
  --config "$rclone_config"
# 使用后删除
rm -f "$rclone_config"
```

### 替代方案（未采用）
- ❌ **永久配置文件**: 存在安全风险，文件可能被泄露
- ❌ **rclone crypt**: 增加复杂度，B2 已有原生加密
- ❌ **Kubernetes Secrets**: 不适用于当前 Docker Compose 架构

## 决策 04: 重试策略

**日期:** 2026-04-06
**状态:** ✅ 已确定

### 决策内容
实现 **指数退避重试**，最多 3 次

### 理由
1. **网络容错**: 临时网络故障可以自动恢复
2. **API 限流**: B2 API 限流时可以等待后重试
3. **指数退避**: 避免频繁重试导致限流加剧
4. **合理上限**: 3 次重试平衡了成功率和时间成本

### 重试策略
```
第 1 次失败 → 等待 1 秒 → 重试
第 2 次失败 → 等待 2 秒 → 重试
第 3 次失败 → 等待 4 秒 → 重试
第 4 次失败 → 放弃，返回错误
```

### 影响
- 上传失败时不会立即报错
- 总重试时间最多 7 秒（1 + 2 + 4）
- 需要在日志中记录重试次数

### 替代方案（未采用）
- ❌ **固定间隔重试**: 效率不如指数退避
- ❌ **无限重试**: 可能导致长时间阻塞
- ❌ **不重试**: 临时故障会导致失败，用户体验差

## 决策 05: 校验和验证方案

**日期:** 2026-04-06
**状态:** ✅ 已确定

### 决策内容
使用 **rclone check --checksum** 验证上传文件完整性

### 理由
1. **可靠性**: 比较本地和云端文件的校验和
2. **自动化**: rclone 自动比较，无需手动计算
3. **快速**: 只比对元数据，不下载文件
4. **详细报告**: 输出不匹配文件的详细信息

### 实现方案
```bash
# 上传后验证
rclone check /local/path b2remote:bucket/path \
  --one-way \
  --combined checksum-report.txt

# 检查退出码
if [[ $? -ne 0 ]]; then
  log_error "校验和验证失败"
  return $EXIT_VERIFICATION_FAILED
fi
```

### 影响
- 上传后自动验证，确保文件完整
- 验证失败时重试或报错
- 增加上传时间（约 5-10 秒）

### 替代方案（未采用）
- ❌ **手动验证**: 容易遗漏，不可靠
- ❌ **下载后验证**: 浪费带宽和时间
- ❌ **不验证**: 无法发现上传损坏的文件

## 决策 06: 旧备份清理策略

**日期:** 2026-04-06
**状态:** ✅ 已确定

### 决策内容
**应用层清理为主**，B2 Lifecycle Rules 作为兜底

### 理由
1. **灵活性**: 应用层可以自定义清理逻辑（如保留每周第一个备份）
2. **一致性**: 本地和云端使用相同的清理策略
3. **可靠性**: B2 Lifecycle 作为兜底，防止应用层清理失败
4. **成本控制**: 及时清理旧备份，控制存储成本

### 清理策略

**应用层清理（lib/cloud.sh）：**
```bash
cleanup_old_backups_b2() {
  # 删除 7 天前的文件
  rclone delete b2remote:bucket/path \
    --min-age 7d \
    --config "$rclone_config"
}
```

**B2 Lifecycle Rules（兜底）：**
```json
{
  "rules": [
    {
      "fileNamePrefix": "backups/",
      "daysFromHidingToDeleting": 30
    }
  ]
}
```

### 影响
- 需要实现清理函数
- 需要在主脚本中调用清理
- B2 Lifecycle 需要手动配置（一次性）

### 替代方案（未采用）
- ❌ **仅 B2 Lifecycle**: 不够灵活，无法实现复杂策略
- ❌ **仅应用层清理**: 可能有遗漏，导致旧文件堆积
- ❌ **手动清理**: 容易忘记，不可靠

## 决策 07: B2 Application Key 权限

**日期:** 2026-04-06
**状态:** ✅ 已确定

### 决策内容
使用 **最小权限原则**，仅授予必要的权限

### 权限配置

**授予的权限：**
```json
{
  "capabilities": [
    "writeFiles",    // 上传备份文件
    "deleteFiles",   // 清理旧备份
    "listFiles"      // 列出备份文件
  ],
  "fileNamePrefix": "backups/",  // 限制到备份目录
  "duration": "valid indefinitely"  // 长期有效
}
```

**不授予的权限：**
```json
{
  "capabilities": [
    // ❌ "readFiles":           不需要读取文件内容
    // ❌ "shareFiles":          不需要公开分享
    // ❌ "writeBuckets":        不需要创建 bucket
    // ❌ "deleteBuckets":       不需要删除 bucket
    // ❌ "listBuckets":         不需要列出所有 bucket
  ]
}
```

### 理由
1. **安全性**: 即使密钥泄露，影响范围最小
2. **合规性**: 符合最小权限原则（SECURITY-02）
3. **审计**: 权限清晰，便于审计
4. **风险控制**: 即使密钥被滥用，只能影响备份目录

### 影响
- 需要在 B2 控制台手动创建 Application Key
- 密钥名称：`backup-system-$(hostname)`
- 密钥需要妥善保管（存储在 .env.backup）

### 替代方案（未采用）
- ❌ **Master Application Key**: 权限过大，安全风险高
- ❌ **所有权限**: 不符合最小权限原则

## 决策 08: 目录结构设计

**日期:** 2026-04-06
**状态:** ✅ 已确定

### 决策内容
使用 **按日期分层** 的目录结构

### 目录结构
```
b2://noda-backups/backups/postgres/
  ├── 2026/
  │   ├── 04/
  │   │   ├── 06/
  │   │   │   ├── keycloak_db_20260406_153000.dump
  │   │   │   ├── noda_prod_20260406_153000.dump
  │   │   │   ├── globals_20260406_153000.sql
  │   │   │   └── metadata_20260406_153000.json
  │   │   ├── 07/
  │   │   └── ...
  │   └── 05/
  └── 2025/
```

### 理由
1. **可读性**: 目录结构清晰，易于查找
2. **可维护性**: 按日期清理旧备份（删除整个日期目录）
3. **兼容性**: 与本地备份目录结构一致
4. **扩展性**: 可以轻松添加新的年份/月份

### 影响
- 上传路径：`b2remote:$B2_BUCKET_NAME/$B2_PATH/YYYY/MM/DD/`
- 清理路径：`b2remote:$B2_BUCKET_NAME/$B2_PATH/YYYY/MM/DD/` --min-age 7d

### 替代方案（未采用）
- ❌ **扁平结构**: 文件太多，难以管理
- ❌ **按数据库分类**: 不如按日期直观

## 决策 09: 错误处理策略

**日期:** 2026-04-06
**状态:** ✅ 已确定

### 决策内容
**上传失败时保留本地备份**，不删除

### 理由
1. **数据安全**: 本地备份是最后一道防线
2. **可恢复性**: 即使上传失败，本地还有备份
3. **可重试**: 可以稍后手动重试上传
4. **符合直觉**: 失败不应该删除数据

### 行为定义
```bash
# 上传失败时
if ! upload_to_b2; then
  log_error "云上传失败，但本地备份已保留"
  log_error "本地备份路径: $backup_dir"
  return $EXIT_CLOUD_UPLOAD_FAILED
fi

# 上传成功后才清理旧备份
cleanup_old_backups_local
cleanup_old_backups_b2
```

### 影响
- 上传失败时会占用本地磁盘空间
- 需要监控磁盘空间
- 需要告警通知上传失败

### 替代方案（未采用）
- ❌ **上传失败时删除本地备份**: 可能导致数据丢失
- ❌ **上传失败时立即重试**: 可能导致长时间阻塞

## 决策 10: 上传顺序

**日期:** 2026-04-06
**状态:** ✅ 已确定

### 决策内容
**先上传元数据，再上传备份文件**

### 上传顺序
```
1. metadata_*.json           # 元数据（包含校验和）
2. globals_*.sql             # 全局对象
3. keycloak_db_*.dump        # 数据库备份
4. noda_prod_*.dump          # 数据库备份
```

### 理由
1. **校验和优先**: 元数据包含校验和，先上传可以用于验证
2. **全局对象优先**: 全局对象通常较小，先上传可以快速完成
3. **可恢复性**: 即使部分备份上传失败，元数据也已保存

### 影响
- 需要控制上传顺序
- 元数据上传失败时，整个流程失败

### 替代方案（未采用）
- ❌ **随机上传**: 不可控，元数据可能最后上传
- ❌ **按文件大小排序**: 小文件优先，但元数据不一定最小

## 决策 11: 并发上传

**日期:** 2026-04-06
**状态:** ✅ 已确定

### 决策内容
**使用 rclone 内部并发**，不在应用层并发

### 配置
```bash
rclone copy ... \
  --transfers 4 \     # 4 个文件并发上传
  --checkers 8        # 8 个并发检查
```

### 理由
1. **简单可靠**: rclone 已实现并发，无需重复造轮
2. **避免限流**: 适度的并发不会触发 B2 API 限流
3. **性能优化**: 4 个并发通常能达到带宽上限
4. **稳定性**: 过多的并发可能导致网络不稳定

### 影响
- rclone 自动管理并发连接
- 不需要应用层实现并发逻辑

### 替代方案（未采用）
- ❌ **应用层并发**: 增加复杂度，可能导致竞态条件
- ❌ **单文件上传**: 速度慢，浪费时间
